## 1. 目标与范围

### 目标
在 `resume/site-crawler` 项目中，将首页（`/`）替换为 Google Stitch 生成的落地页视觉与内容，作为第一阶段“完整替换”版本，为后续内容共创与终版优化打基础。

### 范围内
- 仅替换首页路由展示。
- 使用组件化方式还原 Stitch 页面（方案 B）。
- 首屏保留 URL 输入与 CTA，并可跳转到 `/crawl`。

### 范围外
- 不改动 `/crawl`、`/history`、`/auth/*` 页面内部逻辑。
- 不改动 Worker 接口契约。
- 不在本阶段进行终版文案和 SEO 策略重写。

---

## 2. 设计决策

### 决策 A：只覆盖首页
**结论**：仅覆盖 `app/src/routes/index.tsx` 对应的首页组合内容。
**原因**：用户明确要求“只覆盖首页”，避免影响既有业务流程。

### 决策 B：组件化还原（而非单文件直贴）
**结论**：按首页区块拆分组件并组合。
**原因**：后续将与用户迭代页面内容，组件化能降低修改成本、提升可维护性。

### 决策 C：首轮以视觉与结构一致性优先
**结论**：先 1:1 还原 Stitch 页面主结构和视觉层级。
**原因**：先快速建立基线版本，再进行终版内容优化。

## 3. 页面架构

## 3.1 路由层
- 文件：`app/src/routes/index.tsx`
- 职责：只负责组装首页组件，不承载复杂业务逻辑。

## 3.2 组件层（首页专用）
建议新增或替换为以下区块组件（命名可与现有风格对齐）：

1. `LandingHeader`
2. `LandingHero`
3. `LandingBenefits`
4. `LandingUseCases`
5. `LandingTechnicalSection`
6. `LandingIntegrations`
7. `LandingTestimonials`
8. `LandingPricing`
9. `LandingFooterCta`

> 说明：若项目已有可复用组件（如 `NavBar`、`Footer`），可复用其壳层能力，但不做跨页面大规模重构。

## 3.3 样式策略
- 基于项目现有 Tailwind 体系实现。
- 不引入新 UI 组件库。
- 将 Stitch 中必须保留的少量特殊样式（如 hero pattern）以最小粒度落到组件内样式或全局可控样式中。

---

## 4. 交互与数据流

## 4.1 Hero 输入与 CTA
- 用户在首页输入 URL。
- 点击 CTA 后执行基础校验：
  - 非空
  - URL 基本格式合法
- 校验通过后跳转至 `/crawl`，并附带 URL 参数（如 query）。

## 4.2 静态展示区块
- Use Cases / Technical / Testimonials / Pricing 等区块在本阶段以静态内容还原为主。
- 不接入新后端数据源。

---

## 5. 错误处理

仅处理首页边界输入：
- 空输入：提示用户输入 URL。
- 非法 URL：提示用户输入有效 URL（不触发跳转）。

不新增全局错误边界或远程容错逻辑，避免超范围扩展。

---

## 6. 验收标准

### 功能验收
1. 访问 `/` 能看到 Stitch 风格首页，结构与视觉层级一致。
2. 输入框可输入内容，CTA 可触发。
3. 合法 URL 可跳转到 `/crawl` 并携带参数。
4. 非法输入有前端提示。
5. `/crawl`、`/history`、`/auth/*` 行为不受影响。

### 工程验收
1. `app` 构建通过。
2. 首页与跳转路径可在本地正常访问。
3. 修改范围聚焦首页相关文件，无额外功能扩展。

---

## 7. 风险与缓解

1. **风险：Stitch 导出 HTML 含外部依赖（CDN 脚本/字体）与项目风格冲突**
   **缓解**：转换为项目内 Tailwind 类实现，避免直接依赖导出页 runtime。

2. **风险：首页模块过大导致后续迭代困难**
   **缓解**：首轮即按区块组件化拆分，保持单一职责。

3. **风险：首轮完全替换影响原首页信息架构**
   **缓解**：本阶段只做替换基线；下一阶段按用户反馈做内容终版。
